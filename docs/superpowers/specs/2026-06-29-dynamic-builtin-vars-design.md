# Встроенные (динамические) подстановки `{{$guid}}` и др. — дизайн

**Дата:** 2026-06-29 · **Статус:** утверждён (брейншторм в сессии)

## Проблема

В шаблонах (`{{var}}`) можно сослаться только на пользовательские переменные —
окружения и коллекции. Нет способа подставить сгенерированное на лету значение:
свежий GUID для идемпотентного request-id, текущий timestamp, случайное число.
Сейчас для этого приходится вручную копировать guid из внешнего инструмента в
тело/заголовок на каждый вызов.

Нужны **встроенные подстановки** (aka dynamic variables, как `{{$guid}}` /
`{{$timestamp}}` в Postman/Insomnia): зарезервированные имена, которые при
отправке заменяются сгенерированным значением.

## Набор (6 билтинов, малый курируемый)

| Токен | Значение | Пример |
|---|---|---|
| `{{$guid}}` | v4 UUID (random), lowercase, hyphenated | `7c9e6f82-3b1d-4a2e-9f44-1a0c8e5d2b67` |
| `{{$guid7}}` | v7 UUID (time-ordered) | `019063a2-7c1d-7e3a-9f44-1a0c8e5d2b67` |
| `{{$timestamp}}` | Unix-время, **секунды** | `1782489296` |
| `{{$unixMs}}` | Unix-время, **миллисекунды** | `1782489296123` |
| `{{$isoTimestamp}}` | ISO-8601 UTC (`…Z`) | `2026-06-29T12:34:56Z` |
| `{{$randomInt}}` | целое **0–1000** (паритет Postman) | `742` |

Без параметризации (`{{$randomInt:1:100}}`, кастомных date-форматов) — YAGNI,
отмеченная точка расширения.

## Семантика (зафиксировано в брейншторме)

1. **Синтаксис — `$`-префикс.** Грамматика `VAR_RE = \{\{([^{}]+)\}\}` уже
   пропускает `$` (это не брейс), поэтому **regex не меняется** ни в ядре, ни во
   фронт-зеркале (`hasVars`). «Билтин» = имя из известного набора, начинающееся
   с `$`. Префикс делает коллизию с пользовательскими именами практически
   невозможной.
2. **Приоритет — пользовательская переменная побеждает.** Если пользователь
   завёл env/collection-переменную с именем `$guid`, выигрывает она. Достигается
   **само собой**: `substitute_once` сначала подставляет определённые переменные;
   билтином классифицируется только то, что **осталось** неподставленным после
   прохода. Генераторы — fallback-тир.
3. **Каждое вхождение — своё значение** (Postman-семантика). Два `{{$guid}}` (в
   теле и в метаданных) дают два разных GUID.
4. **Превью — метка + описание, без значения** (Вариант A). В адресной строке,
   редакторе тела и `VarHighlightInput` билтин-токен подсвечивается «динамическим»
   стилем (accent) и показывает тултип-описание (например «v4 GUID · генерируется
   при отправке»), но **не** конкретное значение. Ядро остаётся
   детерминированным; превью не мигает; пользователь видит честное «значение
   появится при Send».

## Архитектура (Подход 1: распознавание в ядре + генерация в команде инвока)

Ключевой факт пути отправки: `grpc_invoke_oneshot` получает `request_json` и
`metadata` **уже** с подставленными пользовательскими переменными — фронт
резолвит их через IPC `vars_resolve` (`resolveStepTemplates` в
`src/features/workflow/resolve.ts`) ещё до вызова. То есть команда инвока —
единственная точка на пути Send, которая видит финальные строки прямо перед
набором gRPC. Это и есть место генерации.

`vars_resolve` обслуживает **и** превью, **и** юзер-вар-часть Send (один IPC),
поэтому генерацию туда класть нельзя (превью бы мигало). Разделение:

- **Фаза 1 — `vars_resolve` (превью + юзер-вары Send): чистая.** Резолвит
  пользовательские переменные, **узнаёт** билтины (не подставляет, не флагует),
  отдаёт их новым полем отчёта.
- **Фаза 2 — `grpc_invoke_oneshot` (только Send): генерация.** `expand_builtins`
  с реальным генератором над телом и метаданными, один раз, перед `invoke_unary`.

### 1. Ядро — новый модуль `crates/handshaker-core/src/vars/builtins.rs`

```rust
/// Единственный источник истины «что такое билтин».
pub fn is_builtin(name: &str) -> bool;        // имя ∈ известный набор ($-префикс)

/// Инжектируемая генерация — ядро остаётся тестируемым/детерминированным.
pub trait BuiltinGenerator {
    /// `None`, если имя не билтин (вызывающий оставляет литерал).
    fn generate(&self, name: &str) -> Option<String>;
}

/// Прод-генератор: uuid v4/v7, SystemTime, RNG.
pub struct SystemBuiltins;
impl BuiltinGenerator for SystemBuiltins { /* … */ }

/// Чистая замена КАЖДОГО вхождения `{{$known}}` свежим значением (per-occurrence).
/// Неизвестные/не-билтин `{{…}}` остаются нетронутыми.
pub fn expand_builtins(s: &str, gen: &impl BuiltinGenerator) -> String;
```

Набор имён и генерация:

| Имя | Генерация (прод) |
|---|---|
| `$guid` | `uuid::Uuid::new_v4()` (нужна фича `v4`) |
| `$guid7` | `uuid::Uuid::now_v7()` (фича `v7` уже включена) |
| `$timestamp` | `SystemTime::now().duration_since(UNIX_EPOCH)?.as_secs()` |
| `$unixMs` | `… .as_millis()` |
| `$isoTimestamp` | ISO-8601 UTC из текущего времени (см. «Зависимости») |
| `$randomInt` | случайное `0..=1000` (см. «Зависимости») |

### 2. Ядро — распознавание в `crates/handshaker-core/src/vars/mod.rs`

`resolve_template_with_diagnostics` и `resolve_string` учатся **узнавать**
билтины среди оставшихся после прохода имён:

- В `ResolutionReport` добавляется поле:
  ```rust
  /// Узнанные билтин-токены ($-префикс, известный набор), в порядке первого
  /// появления, дедуп. Не подставлены — остаются литералом `{{$name}}`.
  pub dynamic_vars: Vec<String>,
  ```
- При классификации остатка (`collect_unresolved`): если `is_builtin(name)` →
  в `dynamic_vars`; иначе → в `unresolved_vars` (как сейчас). Неизвестное `$foo`
  (не из набора) → обычный unresolved (красный).
- `resolve_string` (strict) на билтине **не** падает: оставляет литерал (его
  раскроет Фаза 2 на реальном Send; на прочих путях — см. «Вне scope»).
- **Циклодетектор не трогаем:** билтины не «меняются» проходом, петлю не крутят
  (`converged` остаётся истинным, если остались только билтины), в
  `unresolved_vars` не попадают → `detect_cycle` их не видит.

### 3. Граница IPC (`src-tauri`)

- `ResolutionReportIpc` (`src-tauri/src/ipc/vars.rs`) += `dynamic_vars: Vec<String>`
  с маппингом из `ResolutionReport`. Регенерация `src/ipc/bindings.ts`
  (трекается в git — коммитить с изменением IPC).
- `grpc_invoke_oneshot` (`src-tauri/src/commands/grpc.rs`): перед `invoke_unary`
  прогнать `expand_builtins(&SystemBuiltins, …)` над `request.request_json` и
  каждым **значением** `request.metadata`. **Единственная** точка генерации.
  Ключи метаданных не трогаем.

### 4. Фронтенд (`src`)

- **Превью (Вариант A).** Поверхности, рисующие подсветку по отчёту резолва
  (адресная строка, редактор тела, `VarHighlightInput`), используют новое
  `dynamic_vars`: токен из этого списка красится «динамическим» стилем (accent,
  отличным от обычного резолв-success и от красного unresolved) и показывает
  тултип-описание. Описания токенов — в `src/lib/messages.ts` (правило
  ui-strings), например `settings`-независимый namespace `vars.builtins`.
- **Автокомплит.** Набор билтинов добавляется кандидатами при наборе `{{$` в обе
  поверхности: провайдер тела (`src/features/bodyview/completion.ts`) и
  listbox-дропдаун `VarHighlightInput`. `VarCandidate` (`src/features/vars/
  candidates.ts`) расширяется `origin: "env" | "collection" | "builtin"`;
  билтин-кандидат показывает имя + описание + тег «dynamic». Вставляет
  `{{$name}}`.
- **Набор+описания — параллельный фронт-const** (имя → человекочитаемое описание),
  синхронизируемый с ядровым `is_builtin`. Это сознательное дублирование, как уже
  существующее для грамматики `VAR_RE` (см. память
  `project_var_grammar_core_frontend_sync`); задокументировать комментарием с
  взаимной ссылкой.
- **Send-путь не меняется.** `resolveStepTemplates`/`sendStep` шлют `.resolved`
  (билтины ещё литералом) в `grpc_invoke_oneshot`, который их раскрывает. Гейт
  «unresolved» билтины уже не флагует (их нет в `unresolved_vars`).

## Поток данных (Send, пример)

Тело `{"id": "{{$guid}}", "ts": "{{$isoTimestamp}}", "host": "{{api-host}}"}`,
активное окружение даёт `api-host = api.dev`:

1. `resolveStepTemplates` → IPC `vars_resolve` → Фаза 1: подставляет `api-host`,
   узнаёт `$guid`/`$isoTimestamp`. Возврат:
   `resolved = {"id":"{{$guid}}","ts":"{{$isoTimestamp}}","host":"api.dev"}`,
   `unresolved_vars=[]`, `dynamic_vars=["$guid","$isoTimestamp"]`,
   `cycle_chain=None`.
2. Гейт `ok` (нет unresolved/cycle) → `grpc_invoke_oneshot(request_json=resolved)`.
3. Фаза 2: `expand_builtins` → `{"id":"7c9e…","ts":"2026-06-29T12:34:56Z",
   "host":"api.dev"}` → `invoke_unary`.

## Вне scope (явные follow-up'ы)

- **Билтины в auth-полях, адресе/host, reflection/skeleton.** Эти пути идут через
  `resolve_string` / `resolveAddressSafe`, которые билтин лишь **узнают** и
  оставляют литералом — **не падают**, но и **не раскрывают** (`$guid` в host
  бессмыслен; в auth — редкий кейс). Раскрытие там — отдельный follow-up.
- **Параметризация** (`$randomInt:min:max`, форматы даты).
- **Пер-запросная стабильность** (`{{$guid}}` одинаковый в теле и заголовке) —
  сознательно отвергнута в пользу Postman-семантики «каждое своё».

## Зависимости (за трейтом `BuiltinGenerator`; финал — в плане, проверить по docs)

- `uuid`: включить фичу **`v4`** (сейчас `["v7","serde"]`) для `$guid`; `v7` уже
  есть.
- `$timestamp`/`$unixMs`: `std::time::SystemTime` — без новых крейтов.
- `$isoTimestamp`: рекомендуется крейт **`time`** (лёгкий, `OffsetDateTime::
  now_utc()` + Rfc3339-форматирование) — либо ручной civil-date алгоритм
  (dep-free, но больше кода/тестов). Решение в плане.
- `$randomInt`: **`rand`** (`gen_range(0..=1000)`) либо прямой `getrandom` (уже
  транзитивно через `uuid`). Решение в плане.
- На TLS-унификацию (ring vs aws-lc, см. `project_reqwest_tls_specta_gotchas`)
  эти leaf-крейты не влияют — проверить `Cargo.lock` после добавления.

## Тестирование и гейт

- **Ядро (`cargo test --workspace`):** `is_builtin` (известные/неизвестные
  `$`-имена); `expand_builtins` per-occurrence (два `$guid` → разные значения)
  через детерминированный фейк-генератор; каждый прод-генератор (формат guid v4/v7,
  unix-сек/мс монотонность, ISO-форма `…Z`, диапазон randomInt); `resolve_*` не
  флагует билтин в `unresolved_vars`, кладёт в `dynamic_vars`; приоритет
  (пользовательская `$guid` побеждает); отсутствие ложного цикла.
- **IPC (`cargo test` в `src-tauri`):** `grpc_invoke_oneshot`-уровневый юнит, что
  тело и значения метаданных раскрываются (через фейк/детерминированную проверку
  формы); `ResolutionReportIpc` несёт `dynamic_vars`.
- **Фронт (vitest):** `buildVarCandidates` включает билтины при `{{$`; превью
  красит токены из `dynamic_vars` динамическим стилем; тултип-описания из
  `messages.ts`.
- **Прочее:** `pnpm lint` (tsc) · `pnpm build` (tsc+vite) · `bindings no-drift`.
- **Live (WebView2):** `{{$guid}}`/`{{$guid7}}` в теле → Send → в логе/ответе
  валидные UUID, два вхождения разные; `{{$timestamp}}`/`{{$isoTimestamp}}` →
  текущее время; `{{$randomInt}}` в диапазоне; превью показывает метку+тултип, не
  значение; автокомплит предлагает билтины на `{{$`; неизвестное `{{$foo}}` —
  красное.
