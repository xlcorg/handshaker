# Единый просмотр тела запроса и тела ответа

**Дата:** 2026-06-06
**Статус:** дизайн (ожидает ревью пользователя → плана реализации)
**Тип:** UI/архитектурный рефакторинг рендера двух JSON-тел (фронтенд `src`)
**Ветка:** `redesign/workflow-ui-spec-plans`

> Предшественник — Postman-style рендер ответа
> (`archive/2026-06-04-postman-style-json-response-render-design.md`).
> Тот спек ввёл кастомное виртуализированное дерево ответа; **этот спек его
> заменяет** на общий Monaco-вьювер, объединяя тело запроса и тело ответа под
> единый функционал и вид. Бэкенд и IPC — без изменений.

---

## 1. Контекст и проблема

Сейчас две панели тела реализованы по-разному и расходятся в возможностях:

- **Тело запроса** — `src/features/invoke/BodyEditor.tsx`: Monaco-редактор, язык
  `json-with-vars` (`{{переменные}}`), редактируемый. Есть выделение текста и
  native Ctrl+C/Ctrl+F. **Нет** сворачивания узлов (folding выключен в конфиге) и
  **нет** копирования по даблклику.
- **Тело ответа** — `src/features/response/ResponseBody.tsx` + `json/JsonTreeView.tsx`
  / `JsonLineView.tsx`: кастомное виртуализированное дерево. Есть сворачивание узлов,
  копирование по даблклику, свой поиск, деградация >2MB на кнопку скачивания. Но оно
  **read-only** и **текст в нём выделять нельзя** (своя построчная отрисовка).

Реализации полностью раздельные и сильны в разном: запрос — для редактирования,
ответ — для исследования. Цель — **единый функционал и единый вид** у обеих панелей.

### Целевые свойства (обе панели)

1. Выделение любого текста.
2. Сворачивание/разворачивание узлов «как в Postman» — fold-гаттер редактора.
3. Копирование значения по Ctrl+даблклику.
4. Единый визуальный вид (gutter, подсветка, тема, шрифт).

## 2. Решения (одобрено пользователем)

- **Направление унификации:** обе панели — **Monaco с fold-гаттером** (самый
  Postman-подобный путь). Запрос остаётся редактируемым Monaco, ответ становится
  **read-only Monaco**. Кастомное дерево ответа удаляется.
- **Жест копирования:** **Ctrl+даблклик** (на macOS — ⌘+даблклик) копирует значение.
  Единый жест в обеих панелях. Обычный даблклик везде остаётся нативным
  «выделить слово» — чтобы не ломать редактирование запроса.
- **Семантика копии (богатая, как в текущем ответе):** скаляр-строка → литерал без
  кавычек (полностью); число/bool/null → как есть; объект/массив → compact-JSON всего
  поддерева. Источник — распарсенное дерево, не видимый (усечённый) текст.
- **Поиск:** **native Find Monaco** (Ctrl+F) в обеих панелях. Кастомная панель поиска
  ответа удаляется.
- **Усечение крупных значений (наша фича, которой нет в текущем рендере):**
  - Усекаются **только строковые** значения длиной **> 4096 символов** (≈4 KB; ровно
    как лимит строки CodeMirror в Postman — postman-app-support#4349). Объекты/массивы
    сворачиваются folding'ом; числа/bool/null столько не весят.
  - Отображение: **превью первых ~64 символов** + кликабельный **бейдж размера**
    (UTF-8 байты). Если значение начинается с `data:<mime>;base64,` — в бейдже
    показывается **задекларированный** MIME (`image/png · 248 KB`). Тип НЕ угадывается
    по содержимому — показываем только проверяемое (размер; MIME только если он явно
    написан в значении).
  - **Клик по бейджу** разворачивает **полное** значение инлайн в редакторе —
    **в одну сторону** (сворачивания обратно нет). После разворота бейдж исчезает,
    остаётся обычный полный текст; его можно выделять.
  - Усечение — read-only-фишка **только ответа**. В редактируемом запросе текст всегда
    полный (пользователь его авторит), ничего не прячем.
- **Fold по умолчанию (ответ):** полностью развёрнуто (Postman «Pretty»).
- **Защитный потолок (как в Postman):** по **суммарному размеру тела в байтах**, дефолт
  **50 MB**. Сверх него — плоский read-only-рендер без folding/усечения (последний
  рубеж, аналог Postman «Raw»). Per-value усечение работает всегда и независимо.
  Старая логика degrade→скачивание удаляется.
- **Объём:** только два JSON-тела. Вкладки Metadata/Auth (запрос) и Trailers/Headers
  (ответ) — вне скоупа.

## 3. Архитектура

Один общий Monaco-вьювер для обеих панелей. Различие сводится к двум флагам:

| Флаг          | Запрос            | Ответ                          |
|---------------|-------------------|--------------------------------|
| `readOnly`    | `false`           | `true`                         |
| `render`      | `raw` (как есть)  | `elide` (рендер с усечением)   |

Postman-подобный fold-гаттер, выделение текста и native Find получаем из Monaco
«из коробки». Богатое копирование по Ctrl+даблклику и усечение строим поверх.

### Новый модуль `src/features/bodyview/`

- **`BodyView.tsx`** — тонкая Monaco-обёртка. Конфиг: тема (общая, как сейчас),
  `folding: true`, `readOnly`, native find. Props: `value`, `readOnly`, `render`,
  `onChange?`. Регистрирует dblclick- и badge-click-хендлеры.
- **`jsonRender.ts`** (чистая) — `tree → { text, spans, decorations }`:
  pretty-печать дерева, сбор интервалов значений (`spans`) и усечение крупных
  скаляров (превью + декорация-бейдж). Юнит-тестируема без Monaco.
- **`elide.ts`** (чистая) — порог (4096), формирование превью, размер в байтах,
  извлечение MIME из `data:<mime>;base64,`.
- **`copyValue.ts`** (чистая) — `copyTextFor(node)` (богатая семантика; переезжает из
  `response/json/copyValue.ts`).
- **`spans.ts`** (чистая) — тип `Span = { range, nodeId }`, `binarySearchSpan(spans, pos)`.
- **`jsonTree.ts`** — `parseJsonTree` переезжает сюда из `response/json/`.

`BodyEditor` (запрос) и `ResponseBody` (ответ) становятся **тонкими адаптерами** над
`BodyView`.

## 4. Поток данных

### Ответ (read-only, `render: elide`)

1. Raw JSON → `parseJsonTree` → дерево узлов (каждый хранит `kind`, ключ и **полное**
   raw-значение).
2. `jsonRender(tree)` → `{ text, spans, decorations }`. Для строки > 4096 символов в
   `text` пишется только превью; полное значение остаётся в узле дерева. Бейдж —
   Monaco-декорация (`after`-inlay), **не реальный текст** (не попадает в выделение/Ctrl+C).
3. `model.setValue(text)`; применяются декорации. Дерево + `spans` + карта
   «декорация-бейдж → nodeId» хранятся в `ref` рядом с инстансом.
4. **Ctrl+даблклик:** DOM `dblclick` на `editor.getDomNode()` → проверка `ctrlKey ||
   metaKey` → `preventDefault()` → `editor.getTargetAtClientPoint(x,y).position` →
   `binarySearchSpan(spans, pos)` → `copyTextFor(node)` (из полного значения) →
   `copyToClipboard` + тост.
5. **Клик по бейджу:** точечная правка модели `model.applyEdits([{ range: previewRange,
   text: fullValue }])` (программные правки модели проходят при `readOnly`; readOnly
   блокирует только пользовательский ввод) + удаление декорации-бейджа + пересчёт
   сдвинувшихся `spans`. Разворот **необратимый**.

### Запрос (editable, `render: raw`)

1. `model.value` = текст пользователя как есть (ничего не скрыто, без усечения).
2. **Ctrl+даблклик:** толерантный парс текущего текста (`{{vars}}` → плейсхолдер,
   чтобы JSON парсился) → `spans` на лету (кэш с инвалидацией по `onChange`) →
   `binarySearchSpan` → `copyTextFor`. Если JSON невалиден посреди правки — фолбэк:
   копируем токен по `target.range` через `model.getValueInRange(range)`.

`fold по умолчанию` — ответ разворачивается полностью при установке значения.

## 5. Обработка ошибок и пределы

- **Ответ не парсится в JSON** → показываем сырой текст в read-only Monaco без `spans`
  и усечения; копирование — нативное.
- **Запрос невалиден посреди правки** → фолбэк-копирование токена (см. §4).
- **Сбой clipboard** → тост-ошибка.
- **Whole-body потолок 50 MB** → плоский read-only-рендер без folding/усечения.
  Константа, легко поменять.

## 6. Проверенные факты (Monaco 0.55.1, `node_modules/monaco-editor/monaco.d.ts`)

- `readOnly` — штатная опция редактора.
- `folding: true` — встроенный fold-гаттер (в текущем конфиге запроса выключен).
- `getTargetAtClientPoint(x, y): IMouseTarget | null` (стр. 6382); `IMouseTarget`
  даёт `.position` и `.range` (стр. 5875–5879).
- `IMouseEvent` имеет `detail`, `ctrlKey`, `metaKey`, `altKey` (стр. 502–515).
- `ITextModel.getValueInRange(range)` (стр. 2073); `applyEdits` — штатный API модели.
- Native Find (Ctrl+F) и виртуализация — встроены.

Источники по поведению Postman (для дизайн-решений): лимит строки 4096
([postman-app-support#4349](https://github.com/postmanlabs/postman-app-support/issues/4349)),
whole-body потолок ~50 MB и «Send and Download»
([apidog](https://apidog.com/blog/postman-maximum-response-size-reached/),
[community](https://community.postman.com/t/maximum-response-size/29590)).

## 7. Тестирование (TDD)

Чистые функции (без Monaco):

- `elide`: граница 4096 (4096 — не усекать, 4097 — усекать); превью = первые 64;
  размер в байтах UTF-8 (мультибайт); MIME из `data:image/png;base64,…`; не-`data:`
  строка — без MIME.
- `jsonRender`: корректность `text`, `spans` (диапазоны совпадают с позициями значений),
  декораций; усечение только строк > порога; вложенность/запятые.
- `copyTextFor`: каждый `kind` (строка без кавычек; число/bool/null литерал;
  объект/массив compact-JSON поддерева из полного значения).
- `binarySearchSpan`: попадание в диапазон, границы, промах (вне значений).
- Толерантный парс запроса: валидный JSON; JSON с `{{vars}}`; невалидный → фолбэк.

Компонентные/интеграционные:

- `BodyView` read-only/editable; folding; native find доступен.
- Ctrl+даблклик → мок clipboard получает **полное** значение (в т.ч. усечённое в UI).
- Обычный даблклик остаётся «выделить слово» (clipboard не трогается).
- Клик по бейджу разворачивает полное значение инлайн; повторного сворачивания нет.
- Невалидный ответ → сырой текст без падения.

## 8. Зачистка (мёртвый код после миграции)

Удаляются: `src/features/response/json/JsonTreeView.tsx`, `JsonLineView.tsx`,
`JsonSearchBar.tsx`, `jsonLines.ts`, `jsonSearch.ts`, `degrade.ts`; зависимость ответа
на `@tanstack/react-virtual` (если больше нигде не используется — проверить перед
удалением из `package.json`). `parseJsonTree`/`copyValue` переезжают в `bodyview/`.

## 9. Вне скоупа

- Вкладки Metadata/Auth (запрос), Trailers/Headers (ответ).
- Угадывание типа содержимого по байтам (показываем MIME только если он задекларирован
  в `data:`-URI).
- Сворачивание ранее развёрнутого крупного значения обратно.
