# Спека — двойной клик выделяет значение в теле запроса

**Статус:** 📝 SPEC (одобрена брейнштормом 2026-06-29) — следующий шаг: план.
**Тип:** чистый фронтенд (Monaco / BodyView). Бэкенд / IPC / bindings не трогаются.

## Проблема

В редакторе тела запроса нет удобного способа **заменить значение** в JSON.
Двойной клик в Monaco выделяет только одно «слово», поэтому многословную строку
(`"hello world, foo"`) или длинный токен нельзя выделить одним жестом — приходится
аккуратно тащить выделение вручную. Пользователь хочет: **двойной клик по значению
выделяет значение целиком**, чтобы сразу набрать замену.

## Решение

В редакторе тела **запроса** простой (без модификаторов) двойной клик по значению
выделяет это значение целиком одним жестом.

### Поведение

- **Строка** (`"hello world"`) → выделяется **внутренний текст без кавычек**.
  Набираешь замену — кавычки остаются на месте ⇒ JSON по-прежнему валиден.
  Пустая строка `""` → каретка встаёт между кавычками (готова к набору).
- **number / boolean / null** → выделяется весь токен целиком (кавычек нет).
- **object / array**, **ключ**, структурная пунктуация (`{ } [ ] : ,`) → оверрайда
  нет: остаётся дефолтное word-select Monaco.
- **Ctrl/Cmd + двойной клик** → без изменений (существующий «rich copy»).
  **Shift / Alt + двойной клик** → без изменений (собственные жесты Monaco).
- **Только редактор запроса.** Вьюер ответа — read-only (заменять нечего) — осознанный
  не-гол; копирование в ответе уже есть (Ctrl+двойной клик + контекстное меню).
- Тело в процессе правки и непарсибельно ⇒ `spans` пуст ⇒ двойной клик молча падает
  в дефолт Monaco. Приемлемо.

### Почему именно так

- **Двойной клик** (а не хоткей / Tab / контекстное меню) — самый естественный жест,
  нечему учиться, работает на многословных строках. Выбран на брейнштоме.
- **Внутренний текст у строк** (а не весь токен с кавычками) — оптимально для
  «заменить строку строкой»: выделил `hello world`, набрал `bar` → `"bar"`, валидно.
  Выделение с кавычками потребовало бы перенабирать кавычки. Выбран на брейнштоме.

## Архитектура

Вся инфраструктура локатора значения **уже живёт** в request-режиме `BodyView`:
`live.tree` и `live.spans` пересобираются из сырого текста пользователя на каждой
правке (`parseWithSpans`), а смещения спанов мапятся прямо в позиции Monaco через
`model.getPositionAt`. `attachBodyController` уже ловит двойной клик
(`e.event.detail === 2`) и гейтит существующий Ctrl-clipboard-путь — простой
(без модификаторов) двойной клик встаёт **рядом** тем же механизмом.

### Компоненты

1. **`src/features/bodyview/selectValue.ts`** — новый, чистое тестируемое ядро (TDD):

   ```
   valueSelectionAt(tree: JsonTree, spans: readonly ValueSpan[], offset: number)
     → { start: number; end: number } | null
   ```

   - находит внутренний спан под `offset` (`spanAtOffset`) и его узел;
   - `string`  → `{ start: span.start + 1, end: span.end - 1 }` (внутри кавычек);
   - `number` / `boolean` / `null` → `{ start: span.start, end: span.end }`;
   - `object` / `array` / нет спана → `null`.

   Юнит-покрытие: каждый вид узла, пустая строка (`""` → нулевой диапазон между
   кавычками), клик по ключу (→ `null`, т.к. внутренний спан — объект), клик по
   пунктуации (→ `null`), вложенные значения, отрицательное / экспоненциальное число.

2. **`src/features/bodyview/controller.ts`** — расширить:
   - в `BodyControllerDeps` добавить необязательный
     `onSelectValue?: (range: { start: number; end: number }) => void`;
   - в существующем `onMouseDown` добавить ветку: `detail === 2` **и** ни одного
     модификатора (`!ctrlKey && !metaKey && !altKey && !shiftKey`) → посчитать
     `valueSelectionAt`; если не `null` — `e.event.browserEvent.preventDefault()` +
     `onSelectValue(range)`;
   - в `EditorMouseEventLike.event` (`editorLike.ts`) добавить `altKey: boolean` и
     `shiftKey: boolean` (фейки контроллер-тестов их поставляют).

   Контроллер остаётся свободен от Monaco-selection API (как сейчас он зовёт
   `copyToClipboard`/`onBadgeExpand`) ⇒ юнит-тестируется на смещениях.

3. **`src/features/bodyview/BodyView.tsx`** — только в request-ветке:
   - передать `onSelectValue`, который мапит диапазон смещений в Monaco `Selection`
     (`model.getPositionAt(start/end)`) и зовёт `editor.setSelection(...)`,
     **отложенно** (microtask), чтобы выделение легло **после** того, как Monaco
     отработает собственный двойной word-select (без мигания и гонки порядка);
   - в response-режиме `onSelectValue` не передаётся (поведение ответа не меняется).

### Поток данных

```
двойной клик (без модификаторов)
  → onMouseDown (detail===2, нет модификаторов)
  → offset = model.getOffsetAt(target.position)
  → valueSelectionAt(live.tree, live.spans, offset)
  → onSelectValue(range)
  → (microtask) editor.setSelection(range → Selection)
```

## Тестирование

- **Юнит `selectValue`**: все виды узлов + краевые (пустая строка, клик по ключу,
  пунктуация, вложенность, отрицательное/экспоненциальное число).
- **Юнит `controller`**: фейк-редактор — `onSelectValue` срабатывает с верными
  смещениями на простом двойном клике по значению; **не** срабатывает на
  Ctrl/Cmd/Shift/Alt-кликах и на клике по контейнеру/ключу; существующий
  Ctrl-clipboard-путь не задет.
- **Live WebView2**: двойной клик по длинной строке → выделен весь внутренний текст,
  набор заменяет чисто; number/boolean/null → выделен весь токен; пустая строка →
  каретка между кавычек; Ctrl+двойной клик по-прежнему копирует; в ответе двойной
  клик не изменился.

## Вне scope (YAGNI)

Хоткей выделения значения, Tab-между-значениями, пункт контекстного меню, выделение
**ключа**, выделение значения в режиме ответа.
