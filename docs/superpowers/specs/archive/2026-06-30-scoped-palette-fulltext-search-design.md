## Проблема / цель

В командной палитре (`Ctrl/Cmd+K` / `Ctrl/Cmd+P`) поиск ведёт себя по-разному в
двух режимах:

- **Плоский режим** (коллекция не выбрана): запрос матчится по полному haystack —
  `имя + полный путь сервиса + метод + адрес`.
- **Scope-режим** (после `Tab` в коллекцию): запрос матчится **только по имени**
  сохранённого запроса. Имя сервиса/метод/адрес из поиска выпадают.

Из-за этого, зайдя в коллекцию, нельзя найти метод по имени его gRPC-сервиса —
хотя в плоском режиме это работает. Поведение несогласованное и неожиданное.

**Цель** — в scope-режиме искать по тому же полному тексту, что и в плоском:
`имя + полный путь сервиса + метод + адрес`. Scope-режим становится срезом
плоского по одной коллекции.

## Что уже есть в коде (база для реюза)

- `src/features/catalog/palette.ts` — `rankRequests(query, hits)` уже ранжирует по
  полному haystack `` `${r.name} ${r.service}.${r.method} ${r.address_template}` `` через
  `fuzzy.fuzzyMatch`; пустой query → все хиты по алфавиту имени.
- `src/features/catalog/paletteModel.ts` — чистое ядро `derivePaletteResults`.
  В scope-ветке оно зовёт `rankRequests`, **а затем добавляет лишний пост-фильтр**,
  сужающий результат до совпадений по имени:

  ```ts
  const allHits = col ? rankRequests(query, flattenRequests([col])) : [];
  const hits = query.trim()
    ? allHits.filter((h) => fuzzyMatch(query.trim(), h.request.name).matched)
    : allHits;
  ```

  Именно этот `.filter(...)` и его поясняющий комментарий («match only against
  request name … so that e.g. 'sea' doesn't match 'GetStatus' via the service
  string») — единственное, что делает scope-поиск уже плоского.
- `src/features/catalog/fuzzy.ts` — скорер с бонусами за префикс (+8), начало слова
  (+3), смежность (+5). Благодаря им совпадения по имени всегда ранжируются выше
  совпадений по сервису.
- `CommandPalette.tsx` — тонкий cmdk-компонент; не делает собственной фильтрации,
  только рендерит `result.rows`. Подсветка строится по `indices` (совпадения **по
  имени**) через `matchIndices(query, h.request.name)`.

## Утверждённые решения

| # | Решение | Выбор |
|---|---------|-------|
| 1 | Что искать в scope-режиме | **Полный текст** — `имя + полный путь сервиса + метод + адрес` (как в плоском режиме) |
| 2 | Как реализовать | **Убрать пост-фильтр «только по имени»** в scope-ветке `derivePaletteResults`; scope переиспользует результат `rankRequests` напрямую |
| 3 | Шум по сервису (`sea` → `GetStatus`) | **Принят** — name-совпадения ранжируются выше за счёт бонусов скорера, шум оседает внизу. Граница слова **не** вводится (отвергнута как усложнение) |
| 4 | Подсветка | Без изменений — подсвечивается **только строка имени** (`indices` по `request.name`); совпадение по сервису не подсвечивается, как и в плоском режиме (осознанное прежнее решение) |
| 5 | Пустой query в scope | Без изменений — overview-строка + все запросы коллекции по алфавиту имени |
| 6 | Папки / описания коллекций | **Не добавляются** (на брейншторме не выбраны) |

## Архитектура и модули

### `paletteModel.ts` (единственная правка)

В `derivePaletteResults`, scope-ветка: удалить пост-фильтр по имени и поясняющий
комментарий. Было:

```ts
const allHits = col ? rankRequests(query, flattenRequests([col])) : [];
const hits = query.trim()
  ? allHits.filter((h) => fuzzyMatch(query.trim(), h.request.name).matched)
  : allHits;
```

Станет:

```ts
const hits = col ? rankRequests(query, flattenRequests([col])) : [];
```

Всё остальное в scope-ветке не меняется:

- `requestRows` строятся из `hits` как раньше; `indices: matchIndices(query,
  h.request.name)` остаётся (подсветка по имени, пустая при совпадении по сервису —
  ровно как в плоском режиме);
- overview-строка при пустом query — без изменений;
- заголовок группы `<name> · methods` — без изменений.

Импорт `fuzzyMatch` в `paletteModel.ts` остаётся нужен — `matchIndices` всё ещё им
пользуется (проверить, что линтер не считает его более неиспользуемым: он
используется).

### Незатронутое

- `palette.ts` (`rankRequests` / `rankCollections` / `flattenRequests` /
  `methodLabel`) — без изменений.
- `fuzzy.ts` — без изменений.
- `CommandPalette.tsx` — без изменений (рендер, навигация, key-flow, футер,
  подсветка).
- Плоский режим `derivePaletteResults` — без изменений.
- `bestCollectionMatch`, `completionFor` — без изменений.
- `messages.ts` — новых строк нет.
- Бэкенд / IPC / bindings — не трогаем.

## Состояние и поток

Меняется только **множество совпадений** в scope-режиме. Состояние палитры
(`{ scope, query }`), навигация cmdk (↑/↓/Enter), drill
(`collection → TAB → "." → method`), `Backspace`-выход из scope, открытие
запроса/коллекции — идентичны текущим.

После изменения:

- **Плоский режим** — без изменений.
- **Scope-режим** — `Tab` в коллекцию, далее ввод матчит `имя + сервис + метод +
  адрес` запросов этой коллекции; ранжирование name-first (бонусы скорера); пустой
  ввод — overview + все запросы по алфавиту.

## Тесты (TDD)

- **`paletteModel.test.ts`** — инвертировать существующий тест «drops the overview
  row once the user types a method query» (scope `c1`, query `sea`):
  - было: `ids` содержит `r1` (`Search`), `r2` (`SearchByInn`), **не** содержит
    `r3` (`GetStatus`);
  - станет: `ids` содержит `r1`, `r2` **и** `r3` (т.к. `sea` теперь подпоследовательность
    haystack `GetStatus` через сервис `edo.attorney.v1.Letters` / адрес);
  - добавить проверку **порядка**: `r1`/`r2` (совпадение по имени) идут **раньше**
    `r3` (совпадение только по сервису) — фиксирует «шум оседает внизу».
- **Новый тест scope-режима**: запрос по фрагменту имени сервиса (напр. `letters`
  или `attorney`) в scope `c1` возвращает запросы, у которых совпадение только по
  сервису (имена их не содержат) — подтверждает, что scope теперь ищет по сервису.
- Overview-строка при пустом query, заголовок группы, плоский режим, `completionFor`,
  `bestCollectionMatch` — не должны регрессировать (существующие тесты остаются
  зелёными без правок).

## Вне scope (YAGNI / не выбрано на брейншторме)

- Папки как результаты поиска и матч по именам папок.
- Поиск по описанию коллекции.
- Подсветка подзаголовка `service/method` (совпадение по сервису остаётся
  неподсвеченным — как в плоском режиме).
- Граница слова для совпадений по сервису (`let` → `…Letters`, но не `sea` →
  `GetStatus`) — отвергнута; принят полнотекстовый матч с ранжированием name-first.
- Ранжирование по `use_count` / `last_used_at`, recents на пустом вводе.
- Action-команды (rename/delete/run) — палитра остаётся навигационной.

## Риски / на что смотреть

- **Шум по сервису**: короткий query внутри коллекции теперь может подтянуть методы,
  чьё имя не совпало, но совпал сервис/адрес (общие у всех запросов коллекции). Это
  принятый компромисс; защита — ранжирование name-first. Тест на порядок сторожит,
  что релевантное не тонет.
- **Подсветка пустая при service-only совпадении**: это ожидаемо и согласуется с
  плоским режимом (подсвечивается только имя). Не «баг».
- **Неиспользуемый импорт**: после удаления `.filter(...)` проверить, что `fuzzyMatch`
  ещё импортируется/используется в `paletteModel.ts` (используется в `matchIndices`) —
  иначе `tsc`/линтер выдаст ошибку неиспользуемого импорта.
