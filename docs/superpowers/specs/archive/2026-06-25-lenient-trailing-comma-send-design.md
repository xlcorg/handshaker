**Дата:** 2026-06-25

**Frontend / IPC / bindings:** не трогаются

## Проблема

При Send тела запроса с висячей запятой (`,` непосредственно перед `}` или `]`)
пользователь получает жёсткую ошибку `trailing comma at line N column M`. Это
частая опечатка (удалил поле — забыл запятую).

Корень: тело (`requestJson`) после подстановки `{{var}}` уходит в backend как есть;
[`unary_call`](../../../crates/handshaker-core/src/grpc/invoke/mod.rs) парсит его
строгим `serde_json::Deserializer` → `prost_reflect::DynamicMessage::deserialize`.
serde_json не прощает trailing comma.

При этом в редакторе запятая **не** подсвечивается как ошибка: язык тела — кастомный
`json-with-vars`, без JSON-диагностики. В коде уже есть
[`repairTrailingCommas`](../../../src/features/bodyview/parse.ts), но он применяется
**только** к показу (ghost-хинты, дерево, маркеры контракта), а не к отправке. Итог:
в редакторе тихо, на Send — ошибка.

## Решение (утверждённые решения)

- **Поведение:** тихо принять. Висячая запятая молча убирается **только на проводе**
  перед десериализацией. Текст в редакторе не трогаем, тостов/уведомлений нет.
- **Объём:** прощается **только** настоящая trailing comma (`,` перед `}`/`]`). Любой
  другой невалидный JSON (двойные запятые `[1,2,,]`, sparse `[1,,2]`, прочий мусор) —
  по-прежнему честная ошибка serde_json. Не JSONC: комментарии не поддерживаем.

## Архитектура

Новая чистая функция в ядре (`crates/handshaker-core/src/grpc/invoke/lenient.rs`):

```rust
fn strip_trailing_commas(input: &str) -> Cow<'_, str>
```

Один проход слева-направо, **string-aware**:

- Идём по символам, отслеживая состояние «внутри строкового литерала?» + escape (`\`).
- Вне строки встретили `,` → заглядываем вперёд, пропуская JSON-whitespace
  (` `, `\t`, `\r`, `\n`); если следующий значимый символ — `}` или `]`, запятую
  **выкидываем**, иначе оставляем.
- Внутри строки всё копируется как есть (включая escape-последовательности) ⇒
  строковое значение вида `"x, ]"` не портится.
- На входе без висячих запятых возвращается `Cow::Borrowed` (нет аллокации).

Интеграция в `unary_call`, прямо перед существующим парсингом:

```rust
let cleaned = strip_trailing_commas(request_json);
let mut deserializer = serde_json::Deserializer::from_str(&cleaned);
// ... DynamicMessage::deserialize ...
// последующий deserializer.end() (ловит trailing junk) остаётся без изменений
```

Один call-site — это единственный живой путь отправки (streaming в MVP не
поддерживается; `invoke/skeleton.rs` десериализует только в тестах).

### Почему так, а не иначе

- **Текстовый скраб, без `JSON.parse`/ре-сериализации** — режем только символы
  запятых в сыром тексте, поэтому точность `int64`-чисел и форматирование тела
  сохраняются (ре-сериализация через число потеряла бы точность больших int64).
- **В ядре, а не во фронте** — ошибка рождается в core, это единый чок-пойнт wire;
  чистый Rust удобно покрыть TDD. Существующий фронтовый `repairTrailingCommas` для
  wire **не годится**: голый regex `,(?=\s*[}\]])` испортит строковое значение
  `"x, ]"` (запятая внутри строки попадёт под замену). Поэтому нужен отдельный
  string-aware скраб, и логичнее ему жить в core.
- **Редактор не трогаем** (выбранное «тихое» поведение) — никакой правки каретки/
  контента пользователя на Send.

## Поведение на границах

- `{"a":1,}` → `{"a":1}` ✓
- `[1,2,]` → `[1,2]` ✓
- вложенность `{"a":[1,2,],}` → обе запятые убраны ✓
- строка-гард `{"a":"x, ]"}` → **не меняется** (запятая внутри строки) ✓
- escape в строке `{"a":"he said \", ]"}` → `\"` не закрывает строку, запятая
  остаётся ✓
- `[1,2,,]` → один проход убирает только запятую прямо перед `]` ⇒ остаётся
  `[1,2,]`, что всё ещё ошибка serde_json. **Ожидаемое поведение**: прощаем только
  одну настоящую trailing comma на контейнер, двойные запятые — честная ошибка.

## Тестирование (TDD, Rust в core)

Юнит-тесты на `strip_trailing_commas`:

- `{"a":1,}` → `{"a":1}`
- `[1,2,]` → `[1,2]`
- вложенность `{"a":[1,2,],}` → обе убраны
- строка-гард `{"a":"x, ]"}` → без изменений
- escaped-quote `{"a":"\", ]"}` → без изменений
- no-op `{"a":1}` → возвращает `Cow::Borrowed` / равную строку (нет аллокации)
- `[1,2,,]` → остаётся `[1,2,]` (одна запятая прощена, вторая нет)

Интеграция уровня инвока (зеркало существующего round-trip-теста в
`invoke/skeleton.rs` / `invoke/mod.rs` tests):

- `DynamicMessage::deserialize` через `unary_call`-путь теперь принимает
  `{"field":"v",}` (RED до фикса — падал `trailing comma`, GREEN после).

## Вне объёма (YAGNI)

- JSONC-комментарии (`//`, `/* */`).
- Коллапс двойных/sparse запятых.
- Авто-правка текста в редакторе / тост.
- Изменения frontend / IPC / bindings.
