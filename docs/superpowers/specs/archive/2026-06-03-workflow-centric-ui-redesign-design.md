# Handshaker — Workflow-центричный редизайн UI

**Дата:** 2026-06-03
**Статус:** дизайн (ожидает реализации)
**Тип:** редизайн информационной архитектуры и раскладки фронтенда (React + Tauri + shadcn/ui)

## 1. Контекст и проблема

Текущий UI Handshaker — это классический «Postman для gRPC»: дерево коллекций
слева, вкладки запросов, пара панелей request/response. На реальном основном
сценарии это разваливается.

**Ключевой сценарий (расследование инцидента):**
Инженер дёргает несколько внутренних gRPC-сервисов, чтобы понять, что произошло.
Вызывает метод сервиса A с какими-то параметрами, читает JSON-ответ, **глазами**
находит нужные данные, **руками** копирует их и вызывает метод сервиса B, и так
далее. По ходу набирается ~10 открытых запросов.

**Боли текущей модели:**
- Десяток вкладок — невозможно ориентироваться, теряешь, что уже звал.
- Нет цельной картины расследования; связь между шагами только в голове.
- Поиск метода в огромном каталоге сервисов неудобен.
- Нет понятия «сессия расследования», ответы легко теряются.

**Чего НЕ нужно (явно отвергнуто при проектировании):**
- ❌ Автоматическое связывание данных между вызовами (ссылки вида
  `step1.last_deploy.id`, перепрогон цепочки). Пользователь сам читает ответ и
  копирует значение — он часто заранее не знает, какие поля и в каком формате
  придут.
- ❌ Стриминг (UI только под unary, как и в текущем MVP).

## 2. Доменная модель и термины

Иерархия (важно — уточнена в ходе проектирования):

- **Коллекция** — контейнер, содержащий **k8s-сервисы**.
- **k8s-сервис** — то, что пользователь называет «сервис»: развёрнутый
  endpoint с поднятым gRPC-сервером (адрес `host:port`, напр.
  `payment-api.internal:443`). Может быть **сторонним** (чужой команды).
  Имеет gRPC-контракт, читаемый через **reflection**.
- **Прото-сервис** — proto `service` из контракта. Один k8s-сервис обычно
  экспонирует **несколько** прото-сервисов (`payments.v1.PaymentService`,
  `payments.v1.RefundService`, `grpc.health.v1.Health`).
- **Метод (ручка)** — unary RPC под прото-сервисом. У каждого метода два
  статуса принадлежности:
  - **● в коллекции** — курируемое подмножество, которым команда реально
    пользуется (мелкое).
  - **○ в контракте** — существует в reflection, но не добавлен в коллекцию
    (большой остаток).
- **Workflow** — именованная **упорядоченная последовательность шагов**
  (= одна сессия расследования). Workflow'ов несколько, но обычно мало (1–2).
  Персистентны (как минимум в сессии, и сохраняются между сессиями).
- **Шаг (step)** — один вызов: цель (k8s-сервис + прото-сервис/метод), тело
  запроса, metadata, auth и — после отправки — ответ + статус (gRPC code).
- **Окружение (environment)** — активное окружение **на весь workflow**;
  переключается, влияет на резолв переменных.
- **Избранные сервисы** — помеченные ★ k8s-сервисы для быстрого доступа из
  сайдбара без ⌘K.

## 3. Информационная архитектура

### 3.1 Сайдбар = навигация по коллекции

Сайдбар отдан под **навигацию по коллекции** (не под workflow'ы — те живут в
главной области, см. §3.4). Коллекция курируемая, поэтому её удобно листать
деревом:

- **★ Избранные сервисы** — секция быстрого доступа сверху; раскрывается в
  прото-сервисы → методы.
- **Коллекция** — дерево курируемых `k8s-сервис → прото-сервис → ● метод`.
  Клик по методу → создаёт вызов (открывается в Фокусе, см. §5).
- Сверху — фильтр коллекции.
- **Разделение «курируемое vs весь каталог»:** в сайдбаре листаешь только
  курируемое (● методы — то, чем пользуешься день в день). Полный каталог
  (сотни k8s-сервисов, добавление новых ручек) — через **⌘K** (§3.2) и панель
  сервиса (§3.3). Подсказка внизу сайдбара: «нет нужного — ⌘K».

### 3.2 Поиск: ⌘K «сервис-первый» (двухступенчатый)

Искать по имени метода тяжело (его не помнят); ищут по имени сервиса.

1. **Ступень 1 — сервис.** ⌘K → ввод имени → фильтр по **имени k8s-сервиса**
   в каталоге (сотни сервисов). Результат: сервис + кол-во ручек + адрес.
2. **Ступень 2 — метод.** Выбор сервиса (бредкрамб `PaymentService ›`) →
   список его методов с возможностью дофильтровать → `↵` создаёт вызов.

### 3.3 Панель сервиса (дерево «коллекция ↔ контракт»)

Открывается из ⌘K / избранного. Управление курируемой коллекцией методов:

- Заголовок: `k8s-сервис · payment-api.internal:443`, метка «сторонний · team»,
  «контракт: N мин назад», кнопка **`⟳ Обновить контракт`** (перечитать
  reflection).
- **Дерево**: `k8s-сервис → прото-сервисы → методы`. На методах маркеры
  **● в коллекции** (из них «→ создать вызов») и **○ в контракте**
  (из них «+ в коллекцию»).
- Тулбар: фильтр методов + переключатель **«показать всё из контракта»**
  (снят — видно только ● из коллекции).

### 3.4 Workflows = история вызовов в главной области

Workflow — это **история вызовов**, она **в главной области, перед глазами**
(а не в сайдбаре). Главная область целиком занята активным workflow и
отображается в одном из трёх режимов (§4). Шаги со статус-точками (● OK / ●
error) — это содержимое главной области, а не сайдбара.

- **Переключение между workflow** — **селектор-история в титульной строке**
  (дропдаун: список workflow'ов = историй, выбор активного + создание нового).
  Workflow'ов обычно мало (1–2), поэтому отдельная постоянная панель не нужна;
  достаточно дропдауна.

## 4. Режимы просмотра workflow (3 вида, переключатель в титульной строке)

Workflow можно смотреть в трёх режимах. Переключатель **Лента / Список / Фокус**
живёт в титульной строке.

### 4.1 Лента (Ledger) — ретроспектива
Все шаги сверху вниз на одном холсте. Старые шаги свёрнуты в одну строку
(номер, прото-сервис · метод, статус ✓/✕ с кодом, время). Активный — развёрнут
в request/response. Кнопка «свернуть все» расчищает экран. Максимум наглядности
для обзора всего расследования.

### 4.2 Список (Master-detail) — навигация
Список всех шагов + один выбранный развёрнут крупно справа. Чище, когда шагов
много, но видишь один за раз.

### 4.3 Фокус (Focus) — composing/running одного вызова
**Полноэкранный**, не диалог. Сюда же открывается создание нового вызова.
- Сайдбар сворачивается в **тонкий рельс** со статус-точками шагов (быстрый
  прыжок, не отвлекает).
- **Адресная строка на всю ширину**: 🔒 TLS, имя метода, адрес/прото-сервис,
  статус ответа, ☆, **Send (⌘↵)**.
- **Request | Response занимают всё пространство**, делятся таскаемым
  делителем (можно вертикально сложить) — под большие payload'ы.
- Под-вкладки: Request / Metadata / Auth и Response / Trailers / Headers.
  Нумерация строк, `⧉ копировать всё`, статус-бар снизу (`status 0 · OK · 480B · 53ms`).

## 5. Флоу создания нового вызова

1. Найти метод через ⌘K (или избранный сервис → дерево).
2. Выделить метод → всплывает **сигнатура из контракта** (тип входного и
   выходного сообщения), кнопки `↵ Создать вызов` и `☆ + в коллекцию`.
3. `↵ Создать вызов` → открывается в режиме **Фокус** с **автоматически
   сгенерированным скелетом запроса** (поля input-сообщения из контракта со
   скелетными значениями). **Никакого «черновика» в Ленте** — composing идёт в
   Фокусе, концентрация только на текущем вызове.
4. Заполнить параметры → **Send (⌘↵)** → ответ в той же фокус-области.
5. После отправки вызов **сохраняется шагом** в активном workflow → виден в
   Ленте/Списке среди остальных.
6. **☆ сохранить в коллекцию** — делает разовый ○-метод постоянным ● (в
   следующий раз сразу под рукой).

## 6. Взаимодействия и детали

- **Копирование значения из ответа:** **двойной клик** по значению копирует его
  **целиком** (даже длинное/обрезанное «…»), тост подтверждает. Наведение —
  подсветка строки + tooltip с полным значением. **Без иконки в жёлобе**, без
  обязательного меню. Отдельно — `⧉ копировать всё` для всего ответа.
- **Окружение:** одно активное на workflow, пилюля в титульной строке;
  переключение применяется ко всему workflow.
- **Персистентность:** workflow'ы и ответы **живут только в сессии** (в памяти);
  при закрытии приложения всё очищается. Диск не используется. Тела ответов
  хранятся с **лимитом по размеру** (большие — обрезаются/не сохраняются с
  пометкой). Состояние ведётся автоматически (без явного «сохранить»).
- **Титульная строка:** логотип · селектор workflow · env-пилюля · переключатель
  видов (Лента/Список/Фокус) · ⌘K · настройки.
- **Keyboard-first:** ⌘K (поиск сервиса), ⌘↵ (Send), быстрый прыжок по шагам
  через рельс/список.

## 7. Соответствие исходным болям

| Боль | Решение |
|------|---------|
| Десяток вкладок, не сориентироваться | Workflow-история в главной области: Лента — все шаги со статусами на одном холсте; Список/рельс — быстрый прыжок |
| Нет цельной картины расследования | Workflow как сущность; Лента/Список — ретроспектива |
| Поиск метода в большом каталоге | ⌘K «сервис-первый» + панель-дерево сервиса |
| Чужие сервисы с кучей ручек | Коллекция = курируемое ● подмножество; контракт ○ по кнопке «Обновить» |
| Ответы теряются | Persistent workflow со всеми шагами и ответами |
| Концентрация при вызове | Полноэкранный Фокус, новый вызов открывается туда |
| Не знаю формат запроса | Скелет запроса генерится из контракта |
| Перенос данных между вызовами | Ручной: двойной клик копирует значение (без авто-связей) |

## 8. Вне области (этот редизайн)

- Автосвязывание данных / переменные-ссылки между шагами.
- Стриминг (UI только unary).
- Конкретный визуальный стиль (цвет/типографика) — отдельный трек; здесь
  фиксируется только структура/IA/раскладка. Мокапы нейтрально-тёмные,
  не финальная палитра.

## 9. Декомпозиция реализации (черновой набросок под-планов)

Стратегия — **полная замена** текущего UI, **shell переписывается с нуля**, без
миграции данных. Порядок: первая веха — **Фокус (один вызов + Send)** как ядро.

1. **Ядро: core-типы + IPC + Фокус-вызов.** Сущности (Workflow, Step,
   Collection, Service, Method) в `handshaker-core` + specta-биндинги; reflection
   + скелет-ген; Фокус: создать вызов → скелет → Send → ответ/ошибка.
2. **Каталог и навигация.** Сайдбар-коллекция (избранное + дерево
   `сервис → прото-сервис → ● метод`), добавление сервиса вручную, ⌘K
   «сервис-первый» (fuzzy), панель-дерево сервиса (коллекция ↔ контракт, refresh).
3. **Workflow и режимы.** Workflow как in-memory история; селектор-история в
   титуле; режимы **Лента / Список / Фокус** + переключатель; рельс; удаление и
   drag шагов.
4. **Просмотр ответа.** Кастомный JSON-вьюер (сворачивание, виртуализация
   Postman-style, Ctrl+F), дабл-клик-копирование, рендер ошибок Postman-style.
5. **Env / auth / metadata.** Env на workflow, переменные в адресе/теле/metadata,
   auth и дефолтные metadata на сервис, отмена/таймаут, параллельные Send.

## 10. Решения реализации (интервью 2026-06-03)

**Стратегия и порядок**
- Полная замена текущего UI (без фиче-флага).
- Shell переписываем с нуля (новый стор/каркас).
- Первая веха — Фокус: создать вызов + Send.
- Без миграции старых данных (чистый лист). Старая модель
  Collection→Folder→Request **не переносится**.

**Модель данных и персистентность**
- Workflow'ы — **только в сессии**, в памяти; при закрытии очищаются. Диск не
  используется.
- Тела ответов хранятся **с лимитом по размеру**.
- Схема (Workflow/Step/Collection/Service/Method) — в Rust core + specta-биндинги.
- Состояние ведётся автоматически (без явного сохранения).

**Reflection / контракт**
- Авто-reflection при первом открытии сервиса + кэш на сессию + кнопка «Обновить».
- Нет reflection → понятная ошибка; ручной .proto-импорт **вне области**.
- Скелет запроса: все поля с типовыми дефолтами, с вложенностью.
- Спец-типы: разумные дефолты + подсказки (enum=первое значение, map={}, oneof —
  один вариант + комментарий с альтернативами, timestamp — пример).

**Навигация и ⌘K**
- Каталог k8s-сервисов — пользователь **добавляет вручную** (host:port).
- ⌘K **строго сервис-первый** (метод — только внутри выбранного сервиса).
- Поиск — **fuzzy + ранжирование** (префикс/недавность).
- Полный набор клавиш: ↵ — вызов в текущий workflow, ⌥↵ — в новый, Esc — назад,
  ↑↓ — навигация.

**Поведение workflow / шагов**
- Шаги можно **удалять и перетаскивать** (drag).
- Перевызов **на месте**, ответ заменяется (без истории запусков).
- Правка запроса **мутирует тот же шаг**.
- Стабильные id, отображаемый номер пересчитывается по позиции.

**Режимы просмотра**
- По умолчанию: новый вызов → Фокус, иначе Лента.
- В Ленте развёрнут только активный/последний шаг.
- Выбранный режим запоминается **на каждый workflow**.
- Клик по рельсу в Фокусе переключает фокус на тот шаг (без выхода из Фокуса).

**Просмотр ответа / копирование**
- Кастомный JSON-вьюер со сворачиванием узлов.
- Дабл-клик копирует: строка без кавычек, скаляр как есть, объект/массив —
  компактный JSON.
- Поиск внутри ответа (Ctrl+F).
- Большие ответы — **Postman-style**: виртуализация строк, порог размера с мягкой
  деградацией (пропуск pretty/подсветки, предложение «скачать»).

**Env / auth / metadata**
- Окружение — на workflow, без переопределения на шаге.
- Подстановка `{{var}}` — в адресе, теле и metadata.
- Auth (None / EnvVar / OAuth2) — **только на сервис**.
- Дефолтные metadata — на сервисе, наследуются в шаг (редактируемы).

**Визуал и тех-стек**
- Визуальный стиль — пока текущий shadcn/тёмный; стиль-эксплорейшн — отдельным
  треком.
- Переиспользуем shadcn-примитивы; новые — только фичевые компоненты.
- Тесты: юнит на core + ключевые компоненты; e2e позже.
- Полная клавнавигация + базовый a11y (focus/aria).

**Ошибки и края**
- gRPC-ошибка — **Postman-style**: статус-код + message на виду, decoded details
  + trailing metadata во вкладках.
- Cancel + настраиваемый таймаут.
- Недоступный сервер/TLS — явная диагностика (refused / TLS / DNS / timeout) с
  подсказкой.
- Параллельные Send разрешены (каждый шаг независим).

## 11. Остаётся проработать в плане

- Как новый стор соотносится с текущим Zustand-store (`features/shell`,
  `features/tabs`) — переписываем, но reflection/invoke IPC переиспользуем.
- Точный лимит размера ответа и поведение «скачать» для гигантских payload'ов.
- Точный набор горячих клавиш и их конфликты.
