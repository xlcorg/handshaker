# Send резолвится в ядре; фронт не знает правил резолва

## Контекст

Резолв-пайплайн (переменные → TLS → metadata → auth) существовал дважды: полностью
протестированный `resolve_request` в ядре, который продакшен не вызывал вообще, и
боевое TS-зеркало в `sendStep`/`pickEffectiveAuth`/`resolveAuthHeader`,
синхронизируемое с ядром только комментариями «Mirrors core». Зеркало уже дало
инцидент (16 UNAUTHENTICATED на каждом вызове: живой Send видел только `step.auth`
и терял auth коллекции) — класс багов «UI решил одно, ядро умеет другое».

## Решение

Резолв-пайплайн живёт **только в ядре**. Фронт шлёт в команду Send сырые шаблоны +
ctx-ссылки (`collection_id?`, `env_name?`) и не содержит копий правил резолва и
auth — **даже для отображения**: Auth-таб и снапшот истории спрашивают ядро через
effective-auth запрос, а не вычисляют сами. Auth в ядре расколот на чистый pick
(выбор выигрывающего конфига) и async materialization (заголовок; OAuth2 — через
шов token source), чтобы UI мог спрашивать «кто выиграл» дёшево и синхронно по
смыслу.

## Рассмотренные альтернативы

- **Удалить Rust-копию, оставить TS боевым** — дёшево и честно по принципу
  «тестируй то, что исполняется», но правило auth-цепочки остаётся в TS, а
  env-гейт всё равно зеркалится для UI-индикации: два языка, один инвариант.
- **Половинчато: vars+EnvVar в ядро, OAuth2 на фронте** — пайплайн распилен по
  виду auth; понимание Send хуже, чем до рефакторинга.
- **Резолв на фронте «для отзывчивости»** (в т.ч. локальная копия pick для
  Auth-таба) — это воскрешение зеркала; отзывчивость решается ревизией по смене
  env, а не второй реализацией правила.

## Последствия

- Показ эффективного auth — async-хук с IPC-вызовом (перечитывается по
  `envRevision`/смене origin); это осознанная цена инварианта «одно правило —
  один дом».
- Секреты (`client_secret`, токены) не гуляют через фронт в разрешённом виде.
- 10 тестов `resolve.rs` охраняют боевой путь; TS-тесты зеркала удалены вместе с ним.
- Кто предлагает «маленький локальный резолв в UI» — сначала сюда.
